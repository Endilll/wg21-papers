# Abstract

There are multiple open Core issues against [basic.align]{.sref} ([1211](https://cplusplus.github.io/CWG/issues/1211.html), [2840](https://cplusplus.github.io/CWG/issues/2840.html)) and [dcl.align]{.sref} ([1617](https://cplusplus.github.io/CWG/issues/1617.html), [2223](https://cplusplus.github.io/CWG/issues/2223.html), [3024](https://cplusplus.github.io/CWG/issues/3024.html)). This proposal aims to improve the specification of the aforementined subclauses, resolving at least some of the open issues.

# Implementation divergence

- GCC accepts a defining declaration without an #alignment-specifier# that happens to have the same alignment as specified in other declarations (in violation of [dcl.align]/6) ([Compiler Explorer](https://godbolt.org/z/EfEe4f1zb)):
  ```cpp
  struct alignas(4) A;
  struct A { alignas(4) int i; }; // GCC accepts, Clang, EDG, and MSVC reject
  ```

- Clang considers `alignas(0)` to have no effect for the purposes of [dcl.align]/5, yet considers it to have an effect for the purposes of [dcl.align]/6 ([Compiler Explorer](https://godbolt.org/z/Kchbbhbv9)):
  ```cpp
  struct alignas(0) B1;
  struct B1 {};              // Clang rejects, GCC, EDG, and MSVC accept

  struct alignas(0) B2;
  struct alignas(0) B2 {};   // all accept
  ```

- Clang rejects #alignment-specifier#s on non-static data members of reference type, EDG accepts only if the specified alignment is stricter than the alignment in the absence of #alignment-specifier#, GCC and MSVC accept any #alignment-specifier# (even though non-static data members that are references are not variables) ([CWG3024](https://cplusplus.github.io/CWG/issues/3024.html)) ([Compiler Explorer](https://godbolt.org/z/EGnacosdb)):
  ```cpp
  struct C1 { alignas(2)  const int& r1 = 7; };   // Clang and EDG reject, GCC and MSVC accept
  static_assert(alignof(C1) == 8);                // all accept

  struct C2 { alignas(16) const int& r2 = 7; };   // Clang rejects, GCC, EDG, and MSVC accept
  static_assert(alignof(C2) == 16);               // all accept
  ```

- Clang and GCC prioritize `#pragma pack` over #alignment-specifier#s of non-static data members and their types, MSVC does the opposite, while EDG prioritize `#pragma pack` over #alignment-specifier# of types but not of non-static data members ([Compiler Explorer](https://godbolt.org/z/jc51xcvKs)):
  ```cpp
  struct alignas(8) D1 { int m; };

  #pragma pack(2)
  struct D2 {
    char m1;
    D1 m2;
    char m3;
    alignas(8) int m4;
  };

  static_assert(alignof(D2) == 2); // Clang and GCC accept
  static_assert(alignof(D2) == 8); // EDG, MSVC, and Clang in MSVC compatibility mode accept

  static_assert(sizeof(D2) == 12); // Clang and GCC accept
  static_assert(sizeof(D2) == 24); // EDG accepts; `m4` is at offset of 16
  static_assert(sizeof(D2) == 32); // MSVC and Clang in MSVC compatibility mode accept
  ```

# Issues with status quo wording

- [CWG1211](https://cplusplus.github.io/CWG/issues/1211.html): presumably NAD, because [basic.life]/1 states that lifetime of an object cannot start until "storage with the proper alignment and size for type `T` is obtained". Maybe it's worth harmonizing "proper alignment", "sufficiently aligned", and "meets alignment requirements" across the Standard.
- [CWG1617](https://cplusplus.github.io/CWG/issues/1617.html): [dcl.align]/6 doesn't handle non-defining declarations of a _type_ that specify different alignment, when a definition of that type is not reachable (or they are not rechable from a definition).
- [CWG2223](https://cplusplus.github.io/CWG/issues/2223.html): [dcl.align]/6 doesn't handle non-defining declarations of an _object_ that specify different alignment, when a definition of that object is not reachable (or they are not rechable from a definition).
- [CWG2840](https://cplusplus.github.io/CWG/issues/2840.html): presumably NAD, because [basic.align]/4 already specifies that alignment of `long double`, which is a fundamental type, is a valid alignment
- [CWG3024](https://cplusplus.github.io/CWG/issues/3024.html): presumably NAD, because #alignment-specifiers# can be attached to variables and class declarations, while non-static data memebers that are references are not variables per [basic.pre]/7.
- [basic.align]/1 uses the verb "allocate" to describe alignment, even though this verb is quite tightly coupled with memory allocation and dynamic storage duration.
- [basic.align]/1 repeats [basic.life]/1 saying that creating an object in insufficiently aligned storage is undefined behavior.
- [basic.align]/2 specifies the semantics of `alignof` operator without leaving as much as a note in [expr.align].
- [basic.align]/3, [basic.align]/9, and [dcl.align]/2.2 repeat each other saying that support for extended alignments is implementation-defined, and that a program that needs an unsupported extended alignment is ill-formed.
- [basic.align]/4 and [basic.align]/5 use "valid alignment", but no other wording does.
- [basic.align]/6 doesn't require diagnostics when declarations are in different translation units but one is reachable from another.
- Definition of #alignment-specifier# is separated from [dcl.align].

# Approach

_This section is preliminary in the light of implementation divergence discussed above._

- Alignment is an integral value directly corresponding to layout of objects in memory.
- `alignof` expression reports alignment (which is what all implementations do in the presence of extensions).
- Alignment requirement is a property of types and objects specified by `alignas` and represented as an integral value.
- Two declarations of the same entity shall give it the same alignment.

To enable `#pragma pack` as an extension that a conforming implementation can have ([intro.compliance]/11):

- Correspondence between alignment and alignment requirement is implementation-defined (it seems that the status quo is that alignment requirement specifies the minimum alignment of an object).
- It is implementation-defined whether a given storage meets alignment requirement of a given object (so that the object's lifetime can start even if implementation places it in a storage that doesn't meen alignment requirement).

## Alignment [basic.align]{.sref .unnumbered}

> [1]{.pnum} [Object types have _alignment requirements_ ([basic.fundamental]{.sref}, [basic.compound]{.sref}) which place restrictions on the addresses at which regions of storage an object of that type may be allocated.]{.rm} An _alignment_ is an [implementation-defined]{.rm} integer value [representing the number of bytes between successive addresses at]{.rm} which [describes restrictions on the address of the first byte of a region of storage that]{.add} a given object can [be allocated]{.rm} [occupy]{.add}. [Objects and object types have _alignment requirement_ ([basic.fundamental]{.sref}, [basic.compound]{.sref}), which is a value of type `std::size_t`.]{.add} An object type imposes an alignment requirement on every object of that type; stricter alignment can be requested using the #alignment-specifier# ([dcl.align]{.sref}).
>
> [1a]{.pnum} [Alignment requirement of an entity influences its alignment in an implementation-defined manner. Consequently, it is implementation-defined whether a given region of storage meets alignment requirement of a given object.]{.add}
>
> ::: {.note .add}
> [Attempting to create an object ([intro.object]{.sref}) in storage that does not meet the alignment requirements of the object's type is undefined behavior.]{.rm} [The lifetime of an object cannot start if its storage is not sufficiently aligned ([basic.life]{.sref}), including cases when the object is created by a #new-expression# whose allocation function is a non-allocating form ([expr.new]{.sref}, [new.delete.placement]{.sref}).]{.add}
> :::
>
> [2]{.pnum} A _fundamental alignment [requirement]{.add}_ is [represented by]{.rm} an alignment [requirement that is]{.add} less than or equal to [`alignof(std​::​max_align_t)`.]{.add}
>
> ::: {.note .add}
> `alignof(std​::​max_align_t)` is the greatest alignment [requirement]{.add} supported by the implementation in all contexts[, which is equal to `alignof(std​::​max_align_t)`]{.rm} ([support.types]{.sref}).
> :::
>
> [2a]{.pnum} The alignment required for a type may be different when it is used as the type of a complete object and when it is used as the type of a subobject.
>
> ::: example
> ```cpp
> struct B { long double d; };
> struct D : virtual B { char c; };
> ```
> When `D` is the type of a complete object, it will have a subobject of type `B`, so it must be aligned appropriately for a `long double`. If `D` appears as a subobject of another object that also has `B` as a virtual base class, the `B` subobject might be part of a different subobject, reducing the alignment requirements on the `D` subobject.
> :::
>
> [The result of the `alignof` operator reflects the alignment requirement of the type in the complete-object case.]{.rm}
>
> [3]{.pnum} An _extended alignment [requirement]{.add}_ is [represented by]{.rm} an alignment [requirement]{.add} greater than `alignof(std​::​max_align_t)`. It is implementation-defined whether any extended alignments are supported and the contexts in which they are supported ([dcl.align]{.sref}). A type having an extended alignment requirement is an _over-aligned type_.
>
> ::: note
> Every over-aligned type is [or contains a class type to which extended alignment applies (possibly through a non-static data member)]{.rm} [either specified to have an extended alignment requirement ([dcl.align]{.sref}), or has a subobject _S_ of type _T_, where at least one of _S_ or _T_ is specified to have an extended alignment requirement]{.add}.
> :::
>
> A _new-extended alignment_ is [represented by]{.rm} an alignment [requirement]{.add} greater than `__STDCPP_DEFAULT_NEW_ALIGNMENT__` ([cpp.predefined]{.sref}).
>
> [4]{.pnum} [Alignments are represented as values of the type `std​::​size_t`. Valid alignments include only those values returned by an alignof expression for the fundamental types plus an additional implementation-defined set of values, which may be empty.]{.rm} Every alignment [value]{.rm} [requirement]{.add} shall be a non-negative integral power of two[.]{.rm}[, and shall be in one of the following sets of values:]{.add}
>
> - [4.1]{.pnum} [values returned by an `alignof` expression whose #type-id# designates one of the fundamental types, or]{.add}
> - [4.2]{.pnum} [an implementation-defined set of values.]{.add}
>
> [5]{.pnum} Alignments have an order from _weaker_ to _stronger_ or _stricter_ alignments. Stricter alignments have larger alignment values. An address that satisfies an alignment requirement also satisfies any weaker [valid]{.rm} alignment requirement.
>
> [6]{.pnum} The alignment requirement of a complete type can be queried using an `alignof` expression ([expr.alignof]{.sref}). Furthermore, the narrow character types ([basic.fundamental]{.sref}) shall have the weakest alignment requirement.
>
> ::: note
> The type `unsigned char` can be used as the element type of an array providing aligned storage ([dcl.align]{.sref}).
> :::
>
> [7]{.pnum} Comparing alignments is meaningful and provides the obvious results:
>
> - [7.1]{.pnum} Two alignments are equal when their numeric values are equal.
> - [7.2]{.pnum} Two alignments are different when their numeric values are not equal.
> - [7.3]{.pnum} When an alignment is larger than another it represents a stricter alignment.
>
> [8]{.pnum}
>
> ::: note
> The runtime pointer alignment function ([ptr.align]{.sref}) can be used to obtain an aligned pointer within a buffer; an #alignment-specifier# ([dcl.align]{.sref}) can be used to align storage explicitly.
> :::
>
> [9]{.pnum} [If a request for a specific extended alignment in a specific context is not supported by an implementation, the program is ill-formed.]{.rm}

## Alignof [expr.alignof]{.sref .unnumbered}

> [1]{.pnum} An `alignof` expression yields the alignment [that a hypothetical complete object]{.add} of its operand type [would have]{.add}. The operand shall be a #type-id# representing a complete object type, or an array thereof, or a reference to one of those types.
>
> ::: add
> ::: note
> Alignment can differ between a complete object and a subobject of the same type ([basic.align]{.sref}).
> :::
> :::
>
> [2]{.pnum} The result is a prvalue of type `std​::​size_t`.
>
> ::: note
> An `alignof` expression is an integral constant expression ([expr.const.const]{.sref}). The #typedef-name# `std​::​size_t` is declared in the standard header `<cstddef>` ([cstddef.syn]{.sref}, [support.types.layout]{.sref}).
> :::
>
> [3]{.pnum} When `alignof` is applied to a reference type, the result is the alignment of the referenced type. When `alignof` is applied to an array type, the result is the alignment of the element type.

## Alignment specifier [dcl.align]{.sref .unnumbered}

Change the entire subclause [dcl.align]{.sref} as follows:

> ::: add
> > | _alignment-specifier_
> > > | `alignas` `(` _type-id_ `...`_~opt~_ `)`
> > > | `alignas` `(` _constant-expression_ `...`_~opt~_ `)`
> :::
>
> [1]{.pnum} An #alignment-specifier# may be applied to [a variable or to a class data member, but it shall not be applied to a bit-field, a function parameter, or an #exception-declaration# ([except.handle]{.sref}). An #alignment-specifier# may also be applied to the declaration of a class (in an #elaborated-type-specifier# ([dcl.type.elab]{.sref}) or #class-head# ([class]{.sref}), respectively).]{.rm} [a declaration of]{.add}
>
> - [1.1]{.pnum} [a #class-name# ([class.pre]{.sref}, [dcl.type.elab]{.sref}) or]{.add}
> - [1.2]{.pnum} [a variable, except for]{.add}
>   - [1.2.1]{.pnum} [variables introduced by declarations of function parameters,]{.add}
>   - [1.2.3]{.pnum} [bit-fields, and]{.add}
>   - [1.2.2]{.pnum} [#exception-declaration#s.]{.add}
>
> An #alignment-specifier# with an ellipsis is a pack expansion ([temp.variadic]{.sref}).
>
> [2]{.pnum} When the #alignment-specifier# is of the form `alignas( @#constant-expression#@ )`[:]{.rm} [, the #constant-expression# shall be an integral constant expression that satisfies the requirements of an alignment requirement ([basic.align]{.sref}).]{.add}
>
> - [2.1]{.pnum} [the #constant-expression# shall be an integral constant expression]{.rm}
> - [2.2]{.pnum} [if the constant expression does not evaluate to an alignment value ([basic.align]{.sref}), or evaluates to an extended alignment and the implementation does not support that alignment in the context of the declaration, the program is ill-formed.]{.rm}
>
> [3]{.pnum} An #alignment-specifier# of the form `alignas( @#type-id#@ )` has the same effect as `alignas(​alignof( @#type-id#@ ))`.
>
> [4]{.pnum} The alignment requirement of an entity is the strictest nonzero alignment specified by its #alignment-specifiers#, if any; otherwise, the #alignment-specifiers# have no effect.
>
> [5]{.pnum} The combined effect of all #alignment-specifiers# in a declaration shall not specify an alignment that is less strict than the alignment that would be required for the entity being declared if all #alignment-specifiers# appertaining to that entity were omitted.
>
> ::: example
> ```cpp
> struct alignas(8) S {};
> struct alignas(1) U {
>   S s;
> };  // error: `U` specifies an alignment that is less strict than if the `alignas(1)` were omitted.
> ```
> :::
>
> [6]{.pnum} [If the defining declaration of an entity has an #alignment-specifier#, any non-defining declaration of that entity shall either specify equivalent alignment or have no #alignment-specifier#. Conversely, if any declaration of an entity has an #alignment-specifier#, every defining declaration of that entity shall specify an equivalent alignment. No diagnostic is required if declarations of an entity have different #alignment-specifier#s in different translation units.]{.rm} [If two declarations of an entity does not give it equivalent alignment requirement, the program is ill-formed; no diagnostic is required if neither declaration is reachable from the other.]{.add}
>
> ::: example
> ```cpp
> // Translation unit #1:
> struct S { @[`alignas(4)`]{.add}@ int x; } s, *p = &s;
>
> // Translation unit #2:
> struct alignas(@[`16`]{.rm} [`4`]{.add}@) S;           // ill-formed, no diagnostic required: definition of `S` lacks [alignment]{.rm} [#alignment-specifier#]{.add}
> extern S* p;
> ```
> :::
>
> [7]{.pnum}
>
> ::: example
> An aligned buffer with an alignment requirement of `A` and holding `N` elements of type `T` can be declared as:
> ```cpp
> alignas(T) alignas(A) T buffer[N];
> ```
> Specifying `alignas(T)` ensures that the final requested alignment will not be weaker than `alignof(T)`, and therefore the program will not be ill-formed.
> :::
>
> [8]{.pnum}
>
> ::: example
> ```cpp
> alignas(double) void f();                           // error: alignment applied to function
> alignas(double) unsigned char c[sizeof(double)];    // array of characters, suitably aligned for a `double`
> extern unsigned char c[sizeof(double)];             // no `alignas` necessary
> alignas(float)
>   extern unsigned char c[sizeof(double)];           // error: different alignment in declaration
> ```
> :::
