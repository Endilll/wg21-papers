# Abstract

While looking at [@CWG3103], I got interested how we arrived at status quo, specifically how [[basic.scope.scope]/3.1](https://eel.is/c++draft/basic.scope.scope#3.1) came to be, and this is what I found. One of the conclusions I arrived at is that it doesn't make much sense to overload member functions with explicit object parameter of non-reference type with member functions of any other kind of object parameter with the same type, ignoring references.

# Background

The intent to give member functions with no ref-qualifier special treatment can be tracked all the way to [@N1821]{.title}, which introduced ref-qualifiers:

> The `&` and `&&` suffixes are allowed only on members that allow a cv-qualifier in the same location (e.g., not constructors!). Additionally, neither of these forms can be overloaded with the existing form (the latter retains its binding semantics, including the exception that allows rvalues to be bound).

I think this was caused by the fact that implicit object parameter of a member function with no qualifiers can bind even rvalues, despite being of non-const lvalue reference type ([[over.match.funcs]/5](https://eel.is/c++draft/over.match.funcs#general-5.sentence-3)), which, I think, means the identity conversion and leads to ambiguous calls even without this restriction.

This, and the obvious rule that object parameters of the same type correspond, shape the status quo of the wording.

# Status quo

The table below summarizes whether object parameters of two declarations of member functions correspond (❌) or don't correspond (✅). Relevant wording is [[basic.scope.scope]/3](https://eel.is/c++draft/basic.scope.scope#3) and [[over.match.funcs]/4](https://eel.is/c++draft/over.match.funcs#general-4).

|  |`()`|`() &`|`() &&`|`(this D)`|`(this D&)`|`(this D&&)`|
|:-|:--:|:----:|:-----:|:--------:|:---------:|:----------:|
| **`()`**         |❌||||||
| **`() &`**       |❌|❌|||||
| **`() &&`**      |❌|✅|❌||||
| **`(this D)`**   |❌|✅|✅|❌|||
| **`(this D&)`**  |❌|❌|✅|✅|❌||
| **`(this D&&)`** |❌|✅|❌|✅|✅|❌|

Implementations agree on 18 out of 21 cases ([Compiler Explorer](https://godbolt.org/z/aM963qh8n)):

- GCC and MSVC fully conform;
- EDG has a bug where explicitness of the object parameter by itself is enough to break correspondence despite types being the same;
- Clang sometimes considers `(this) + (this D&)` and `(this) + () &` cases to correspond, but this might be a desirable direction as discussed in [](#proposed-changes) section. This happens when they are written in exactly this lexical order (this quirk has been filed as [llvm/llvm-project#189525](https://github.com/llvm/llvm-project/issues/189525)).

However, when one of the declarations is named by a using-declaration, implementations agree only on 18 out of 36 cases, which includes 15 additional symmetrical cases left blank in the table ([Compiler Explorer](https://godbolt.org/z/fah8aK4c6)):

- MSVC struggles a lot;
- EDG struggles a lot, too, but less than MSVC;
- Clang conforms if using-declaration and declaration of member function are in the "right" lexical order, which is a unique behavior across surveyed implementations;
- GCC fully conforms.

Note that in this latter case correspondence (❌) doesn't translate to program being ill-formed per [[namespace.udecl]/11](https://eel.is/c++draft/namespace.udecl#11).

# Not all ✅ are created equal

The table below summarizes which value category the object argument has to have for a call to be well-formed, focusing on cases with `(this)` overload.

|| By itself<br>(copy+move) | With<br>`(this CopyMove)` | By itself<br>(move-only) | With<br>`(this MoveOnly)` |
|:----|:---:|:----:|:---:|:-----:|
| 1. **`()`**        | Both   | Conflicting | Both   | Conflicting |
| 2. **`() &`**      | Lvalue | Rvalue      | Lvalue | Rvalue      |
| 3. **`() &&`**     | Rvalue | Lvalue      | Rvalue | Neither     |
| 4. **`(this)`**    | Both   | Conflicting | Rvalue | Conflicting |
| 5. **`(this &)`**  | Lvalue | Rvalue      | Lvalue | Rvalue      |
| 6. **`(this &&)`** | Rvalue | Lvalue      | Rvalue | Neither     |

All implementations [agree](https://godbolt.org/z/baPGKKq39), with an exception of Clang, which considers more cases conflicting, as described in the previous section.

Note how adding a `(this)` overload to an existing member function _F_ either flips the set of well-formed calls to its complement, rendering *F* obsolete, because `(this)` is always the best viable function, or outright make any call ill-formed (cases 3 and 6 with move-only type). However, a function that cannot be selected by an overload resolution for a call expression can still be selected by an address of an overload set with the right target:

```cpp
struct CopyMove {
  void f() &;             @[// #1]{.comment}@
  void f(this CopyMove);  @[// #2]{.comment}@

  void g(this CopyMove&); @[// #3]{.comment}@
  void g(this CopyMove);  @[// #4]{.comment}@
};

void test(CopyMove& copymove) {
  copymove.f();   @[// ambiguous]{.comment}@
  CopyMove{}.f(); @[// calls #2]{.comment}@
  void(CopyMove::*p1)() & = &CopyMove::f; @[// selects #1]{.comment}@

  copymove.g();   @[// ambiguous]{.comment}@
  CopyMove{}.g(); @[// calls #4]{.comment}@
  void(*p2)(CopyMove&) = &CopyMove::g;    @[// selects #3]{.comment}@
}
```

While it's possible to get ahold of functions #1 and #3 and call them, it's not clear why they need to be a part of their respective overload sets in the first place.

# Proposed changes

Explicit object parameter of non-reference types should correspond with any other object parameter if they have equivalent types, ignoring references, i.e. it should be ill-formed to have overloads with such object parameters that don't have any other differences. This change will have the following effects:

- Correspondence of member functions with explicit object parameters of non-reference types will be synchronized with implicit member functions with no [ref-qualifier]{.grammarterm}.
- The second example in [@CWG3103] will become ill-formed as requested.
- Clang rejecting `(this) + (this D&)` and `(this) + () &` overloads when they are written in exactly this lexical order, as described above, will become conformant behavior.

Table below summarized correspondence of object parameters with the proposed changes applied. Combinations that are made to correspond by this change are marked as ❌❌.

|  |`()`|`() &`|`() &&`|`(this D)`|`(this D&)`|`(this D&&)`|
|:-|:--:|:----:|:-----:|:--------:|:---------:|:----------:|
| **`()`**         |❌||||||
| **`() &`**       |❌|❌|||||
| **`() &&`**      |❌|✅  |❌  ||||
| **`(this D)`**   |❌|❌❌|❌❌|❌  |||
| **`(this D&)`**  |❌|❌  |✅  |❌❌|❌||
| **`(this D&&)`** |❌|✅  |❌  |❌❌|✅|❌|

# Proposed wording

## Wording option 1

Change [basic.scope.scope]{.sref} paragraph 3 as follows:

> [3]{.pnum} Two non-static member functions have *corresponding object parameters* if
>
> - [3.1]{.pnum} [exactly one is an implicit object member function with no [ref-qualifier]{.grammarterm} and the types of their object parameters ([dcl.fct]), after removing references, are the same, or]{.rm}
> - [3.1]{.pnum} the types of their object parameters [([dcl.fct], [over.match.funcs])]{.add} are equivalent[.]{.rm}[, or]{.add}
> - [3.2]{.pnum} [if one of the functions is]{.add}
>   - [3.2.1]{.pnum} [an implicit object member function with no [ref-qualifier]{.grammarterm} or]{.add}
>   - [3.2.2]{.pnum} [a member function with an explicit object parameter whose type is not a reference type]{.add}
>
>   [the types of their object parameters, after removing references, are equivalent.]{.add}

## Wording option 2

Change [basic.scope.scope]{.sref} paragraph 3 as follows:

> [3]{.pnum} Two non-static member functions have *corresponding object parameters* if
>
> - [3.1]{.pnum} exactly one is an implicit object member function with no [ref-qualifier]{.grammarterm} and the types of their object parameters ([dcl.fct]), after removing references, are the same, or
> - [3.2]{.pnum} [one is a member function with an explicit object parameter whose type is not a reference type, and the types of their object parameters, after removing references, are equivalent, or]{.add}
> - [3.3]{.pnum} the types of their object parameters are equivalent.
